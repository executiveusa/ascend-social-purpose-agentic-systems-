# v0.4 Production Core Upgrade

This release shifts Mission OS from a polished scaffold toward a repeatable production core.

## Core decisions

- Custom frontends are replaceable per nonprofit.
- The backend operating system remains shared and centrally improved.
- ICM stays the agent architecture: numbered folders, `CONTEXT.md` contracts, Layer 3 references, Layer 4 artifacts, and approval gates.
- Rust owns sensitive/durable boundaries: tenant core, public bridge, policy classification, outbox worker, and filesystem-safe ICM runner.
- Next.js remains the fastest layer for public sites and the staff/admin console.

## New layers

```text
services/mission-core-rs      authoritative tenant/CRM/approval core
services/mission-connect-rs   public frontend bridge for custom websites
services/mission-worker-rs    outbox worker and scheduled durable jobs
services/mission-policy-rs    deterministic action-risk classifier
services/mission-icm-rs       filesystem-safe ICM stage runner
packages/mission-sdk-js       JS SDK for any client frontend
packages/tenant-kit           frontend config/theme/llms helpers
missionctl/                   repeatable tenant/deploy control plane
```

## Golden workflow

```text
missionctl tenant create youth-org --org "Youth Org" --region "Seattle / King County"
missionctl frontend scaffold youth-org
missionctl smoke youth-org
npm run verify
```

The frontend receives only tenant config and a public API key. All form submissions, messages, volunteer intake, program applications, and donation intents hit the shared backend bridge.

## Production acceptance criteria

1. A new tenant can be created with one command.
2. Public keys and origin allowlists are generated.
3. A custom frontend can submit contact/volunteer/program forms without custom backend work.
4. The backend converts those submissions into CRM contacts, interactions, and pipeline items.
5. Orange/red actions cannot execute without approval.
6. Approved external work goes through the outbox.
7. ICM outputs are written under the tenant boundary only.
8. `missionctl doctor` and `npm run verify` pass before deployment.
