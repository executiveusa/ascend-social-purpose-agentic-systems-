# Release Manifest — Mission OS v0.6

## Overview

A release manifest is generated by `missionctl bundle release <tenant>` and stored at `handoff/<tenant>/managed/release-manifest.json`. It records the version, bundle metadata, and links to the corresponding deployment release record.

## Manifest fields

```json
{
  "version": "0.6.0",
  "tenant": "demo-pnw",
  "bundle_type": "managed",
  "created_at": "2026-07-01T...",
  "git_sha": "7bf424c",
  "pack_version": "1.0.0",
  "services": {
    "postgres": "postgres:16-alpine",
    "redis": "redis:7-alpine",
    "hermes": "hermes:latest",
    ...
  },
  "released_at": "2026-07-01T...",
  "status": "released",
  "release_id": "rel_<hex>"
}
```

## Lifecycle

1. `missionctl bundle up <tenant> --dry-run` — generates the bundle and writes a `release-manifest.json` template.
2. `missionctl bundle release <tenant>` — sets `released_at`, `status: released`, and `release_id`; creates a `deployment_releases` record (status: `draft`).
3. `missionctl upgrade <tenant> --release <id>` — activates the release record (status: `active`).

## Checksum

The release manifest does not include per-file checksums of the bundle itself (that level of integrity verification is done by `missionctl bundle smoke`). The backup system (`deployment-backup.js`) computes a `checksum_sha256` over all files in the backup directory.

## Artifact registration

`missionctl bundle release` also calls `registerArtifact` to add the manifest to the tenant artifact registry, making it visible in the `/ops/artifacts` ops dashboard view.
