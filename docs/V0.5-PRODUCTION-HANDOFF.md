# Mission OS v0.5 — Production Handoff Pass

## What v0.5 adds

v0.5 is focused on repeatable deployment rather than new dashboard features.

Added:

- Hostinger handoff generator: `missionctl hostinger handoff <tenant>`
- Root `HOSTINGER-VPS-HANDOFF.md` generation
- Tenant-specific handoff bundle under `handoff/<tenant>/`
- Frontend bridge `.env` generator
- Caddy and Docker production templates
- Public bridge idempotency keys
- Public bridge honeypot rejection
- Public form submissions create CRM tasks
- CRM task API
- AdamsReview-lite multi-lens release artifact

## Release posture

This is still not a fully hosted production system until the VPS deployment is run end-to-end, but it is now packaged for that deployment.

## The repeatable client workflow

```bash
node missionctl/missionctl.mjs tenant create client-slug \
  --org "Client Org" \
  --domain "https://client.org" \
  --api "https://api.client.org"

node missionctl/missionctl.mjs frontend scaffold client-slug
node missionctl/missionctl.mjs hostinger handoff client-slug \
  --domain "client.org" \
  --api-domain "api.client.org" \
  --vps-ip "<HOSTINGER_VPS_IP>"
```

The frontend team receives:

- `NEXT_PUBLIC_MISSION_API_URL`
- `NEXT_PUBLIC_MISSION_TENANT`
- `NEXT_PUBLIC_MISSION_PUBLIC_KEY`
- Mission SDK example
- `llms.txt` starter

## Senior developer warning

Do not customize backend code per client. If a client needs a new form or workflow, add it to the shared bridge/contracts and version it. The backend should improve centrally; client customization should happen in the frontend, tenant config, ICM folders, and brand system.
