# Security Checklist — Mission OS v0.6

Use this checklist before every release, merge to main, and production handoff.

## Pre-commit gate (automated)

Run before every commit to main or feature branch:

```bash
node scripts/secret-audit.mjs
node scripts/generated-file-audit.mjs
node scripts/test-discovery-audit.mjs
```

Or use the full verifier:

```bash
node scripts/verify-v06.mjs
```

## Secret hygiene

- [ ] No raw operator keys (`ok_<tenant>_<hex>`) in any tracked file
- [ ] No raw `NEXTAUTH_SECRET`, `WEBUI_SECRET_KEY`, `LITELLM_MASTER_KEY` values in tracked files
- [ ] No provider keys (`sk-...`) in tracked files
- [ ] No `JWT_SECRET`, `POSTGRES_PASSWORD` values (non-placeholder) in tracked files
- [ ] `handoff/*/managed/hermes/env` NOT in `git ls-files`
- [ ] `handoff/*/managed/langfuse/env` NOT in `git ls-files`
- [ ] `handoff/*/managed/litellm/env` NOT in `git ls-files`
- [ ] `handoff/*/managed/open-webui/env` NOT in `git ls-files`
- [ ] `handoff/*/managed/release-manifest.json` NOT in `git ls-files`
- [ ] `handoff/*/managed/.env.managed` NOT in `git ls-files`
- [ ] `mission-data/` NOT in `git ls-files`
- [ ] `backups/` NOT in `git ls-files`

Safe files (may be tracked):
- `*.env.example` — placeholder-only templates
- `*.env.managed.example` — placeholder-only managed bundle templates

## Runtime secret generation rules

1. All runtime-generated env files go to `handoff/<tenant>/managed/` and are gitignored.
2. Operator keys are generated by `missionctl operator-key create` and stored hashed in `mission-data/`.
3. Never copy real key values into tracked template files.
4. If a key must appear in a template, use `<GENERATED_BY_MISSIONCTL>` or `change_me`.

## Tenant isolation

- [ ] Tenant slug validation: only `[a-zA-Z0-9_-]` allowed
- [ ] `DATA_DIR/<tenantId>/` is the only writable path for a tenant
- [ ] Path traversal is blocked in backup/restore (`../` and absolute paths rejected)
- [ ] Operator keys are scoped to a single `tenant_id`
- [ ] No tenant can read another tenant's `mission-data/` directory

## Approval gates (enforced)

These actions require human approval before execution:

- Public publishing (Postiz, social media)
- Donor/youth outreach (Twilio/email)
- Financial submissions (grants, invoices)
- Browser automation on external portals
- Red or orange risk-class agent actions

## Authentication model

- User sessions: SHA-256 hashed, never raw
- Operator keys: prefixed `ok_<tenantId>_`, SHA-256 hashed at rest
- No raw operator keys exposed to client-side JavaScript (enforced by `ops-no-operator-keys-in-client.test.js`)
- Session JWT secret must not be the dev default in production

## Dependency audit

```bash
npm audit --audit-level=high
```

Run before every release. Known accepted risks must be documented in `docs/PRODUCTION-GAPS.md`.
